# incident_type design

Add `incident_type` to `ExtractedArticleData` as `Optional[Literal[...]] = None` near the incident fields in `crime_pipeline/models.py:252`, before weapon fields. `None` is required for backward compatibility with existing DB rows and older extraction JSON.

Recommended values:
`homicide`, `attempted_homicide`, `accident`, `traffic_accident`, `suicide`, `historical_retrospective`, `arrest_only`, `protest`, `other`, `unknown`.

I would add `traffic_accident` to the suggested baseline because traffic deaths are frequent false positives and need different wording from workplace/industrial accidents. I would not add a separate `workplace_accident`; it is covered by `accident`. `arrest_only` should include fraud/cheating/drug/weapon-possession arrests when the article does not describe a homicide or attempted homicide incident. `historical_retrospective` covers anniversary, memorial, statistics, and commemoration articles whose deaths occurred outside the article's current incident.

Relevance policy:
Keep only `homicide` and `attempted_homicide`. Drop all other explicit values with reason `incident_type_<value>`. This fits the homicide dataset while preserving attempted homicide as a follow-up seed; export can still drop non-fatal rows by `victim_outcome`.

Null/missing policy:
Keep legacy behavior when `incident_type` is null or missing. Do not drop solely on null, because old extracted records do not have this field and because LLM omission should not create a silent false negative. The field only becomes decisive when explicitly populated.

Prompt placement:
Put an `INCIDENT TYPE RULE` immediately before the existing `VICTIM OUTCOME RULE` in `crime_pipeline/extraction/prompts.py:35`. It classifies the article's primary current incident before lethality, and the Arabic/Hebrew examples sit next to the existing Arabic/Hebrew outcome vocabulary.
