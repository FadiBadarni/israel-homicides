# Gemini — Discover Phase Research: Pipeline Validation Patterns

## 1. dbt 'Tests vs Models' for Cleanup Passes
In the `crime_pipeline` context, cleanup passes like `sanity_pass` and `reconcile` occupy a hybrid space. While they function as **transformations** (merging cases, fixing attributes), they are fundamentally driven by **validation logic**.

**Recommendation:** Treat cleanup passes as **Inline Transformation Stages**.
In dbt, a "test" that fixes data would be a model. Since these passes mutate the case dictionary to improve quality, they belong in the primary `stages` list of the pipeline. However, they should be implemented as **filtering/mutating middleware**. 
- **Sanity/Quality**: Should be stages that articles/cases must pass through. If a case fails "Sanity," it is flagged or dropped (transformation).
- **Architecture**: Keep them in the `stages` list so they benefit from the existing checkpointing logic in `pipeline.py`.

## 2. Dagster Asset-Checks Pattern
Dagster's `@asset_check` pattern is superior to silent mutation for long-term observability. 

**Recommendation:** Emit **Asset Check Results** as metadata.
Instead of silently dropping a case in `quality_pass.py`, the stage should append a `validation_results` list to the case object.
- **Severity**: Use `WARN` for missing fields (e.g., neighborhood) and `ERROR` for fatal flaws (e.g., victim survived a "homicide").
- **Blocking**: The `Export` stage should explicitly check for `ERROR` level results and skip those cases. This mirrors Dagster’s "skip downstream on fail" behavior while preserving the "bad" data for debugging within the JSON artifact.

## 3. Audit Log Format for Reconcile Merges
When `reconciler.py` merges two cases, the provenance is currently "lost" in the sense that the weaker case just disappears.

**Recommendation:** Add a `reconciliation_provenance` field to the **Canonical Case**.
Writing a side-file (`output/{run_id}_reconcile_audit.jsonl`) introduces a synchronization burden. The React UI benefits most if the case itself knows its history.
- **Schema**:
  ```json
  "reconciliation_provenance": [
    {"merged_from_url": "...", "reason": "name_match", "jaro_score": 0.92}
  ]
  ```
- **Benefit**: The UI can show a "Merged from X sources" badge with a tooltip explaining the match logic, increasing user trust in the AI's "judgement."

## 4. JSON Schema Forward-Compat for React UI
The pipeline is evolving (adding `tier_coverage`, `motive_translations`). The React UI must not crash when reading "Legacy v1" JSONs.

**Recommendation:** **Optional Fields with Sane Null Defaults**.
Schema versioning and migrations are high-overhead for a prototype. 
- **Pattern**: The TypeScript interfaces in `ui/frontend/lib/api.ts` should mark all new fields as optional (`field?: type`). 
- **UI Implementation**: Use optional chaining (`case.motive_translations?.he ?? 'N/A'`). This allows the UI to naturally "degrade" when displaying older runs without requiring a batch-migration of the `output/` directory.

## 5. CLI Flag Pattern
The current `--stage` implementation in `__main__.py` is functional but lacks "negative" control (skipping one stage without listing all others).

**Recommendation:** Extend `--stage` choices and add a `--skip-stage` flag.
Following the `kubectl` and `terraform` patterns:
- **Keep `--stage`**: For targeting specific work (e.g., `--stage export`).
- **Add `--skip-stage`**: A repeatable option (e.g., `--skip-stage sanity --skip-stage quality`).
- **Implementation**: In `__main__.py`, calculate `final_stages = (all_stages | set(stages)) - set(skip_stages)`. This is the most idiomatic way to handle "validation" toggles in Click-based CLIs without cluttering the namespace with dozens of `--no-X` flags.
