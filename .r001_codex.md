# Codex (Round 1) — Pick: G

Pick G: B+C+E, with C implemented as a persisted Stage 2.5 between Fetch and Extract, not as an external side path.

Current flow is Discover -> Fetch -> Extract -> relevance -> Dedup -> Merge -> Sanity -> Quality -> Reconcile -> Export. The new flow should be Discover -> URL/title/date cheap reject -> Fetch -> Triage -> Extract only yes/maybe -> existing relevance filter -> rest unchanged.

B belongs inside/just after Discover. Arab48 already extracts URL path dates and titles into `DiscoveredUrl`; add source-specific negative markers there as an allow-to-fetch decision, but only for high-confidence exclusions like work accidents, anniversary retrospectives, protests, culture/sports. This saves fetch and LLM cost, but it must never be the main recall gate because URL slugs are editorial, not data.

C is the real cost lever. Run a tiny classifier on title + first 200-400 chars after Fetch, persist `triage_status`, `triage_reason`, model, tokens, and prompt version. Extract only `yes` and `maybe`; sample or periodically replay `no` for validation. This keeps the current extraction schema and downstream relevance filter intact. The classifier should be recall-biased: uncertain, attempted shooting, missing death marker, or city ambiguity all become `maybe`.

E is operationally right for refreshes. Daily 7-day windows reduce historical-retrospective noise and match publishing cadence. But E breaks the mental model that one pipeline run equals a complete city-year answer. The checkpoint DAG remains linear per run, but product queries must read accumulated DB state across runs. Cold start still needs a bounded backfill job, using the same G gates.

Cost: brute force is 900 articles * ~5K input = 4.5M input tokens per city, plus about 630K output tokens, roughly the cited $1-3 all-in. With G, 900 triage calls at maybe a few hundred tokens each are cheap, and if only 10-25% reach full extraction, full-extract input falls from 4.5M to ~0.45-1.1M. That is a meaningful reduction without making recall depend on narrow queries.

I would not choose D or F as primary. Category URLs and multi-query search are useful source-specific supplements, but both can silently miss articles. The authoritative homicide decision should remain content-derived.
