## A. CLI shape for stage 4 (--cities backfill)

### Does --cities replace or supplement --query?

Currently `--query` is required unless `--enrich-case` or `--reconcile` is set
(line 295 of `__main__.py`). `--cities` must be a peer alternative, not a
supplement. The mutual-exclusion guard that currently reads:

```python
if not query:
    click.echo("ERROR: --query is required ...", err=True)
    sys.exit(2)
```

...must become a three-way check: fail only when *none* of `--query`,
`--cities`, `--enrich-case`, `--reconcile` are provided.

### Comma-separated list vs file?

Comma-separated list is the right default; the current codebase uses this
pattern for every multi-value flag (`--sources ynet,arab48`, `--stage`
repeatable). Adding a `--cities-file` path variant is low-risk and useful for
30+ city batches but is out of scope for stage 4. Propose:

```
--cities arraba,sakhnin,umm-al-fahm
```

Parsed the same way as `--sources`: `[s.strip() for s in cities.split(",")]`.

### What script should the operator type?

English (transliterated) is the right UX default. The gazetteer already
indexes every city under all three scripts plus aliases. The operator-facing
token is purely a lookup key — it will be matched after `_normalise_key()`
which lowercases and NFC-normalizes. So `arraba`, `Arraba`, `عرابة`, and
`עראבה` all resolve to the same `CityRecord`. However, the operator is most
likely running from a shell; Arabic/Hebrew input requires IME switching and is
error-prone. English should be documented as canonical in `--help` text, with a
note that Arabic/Hebrew are also accepted.

### Per-city run loop: sequential Pipeline.run() calls vs internal multi-query loop?

The right answer is **sequential calls to Pipeline.run()**, one per city, from
`__main__.py`. Do NOT push the loop into `Pipeline.run()`. Reasons:

1. `Pipeline.__init__` sets `self.run_id` and initializes `self.stats` in
   `__init__`, not in `run()`. If you call `run()` multiple times on the same
   instance, `self.stats` accumulates across cities, producing a single
   misleading stats blob, and `self.run_id` stays the same for all cities.

2. `self._media_pipeline = MediaPipeline(self._media_settings)` is created in
   `__init__`. This is actually a *benefit* of looping over `Pipeline`
   instances — the CLIP model gets loaded once per process (see section D),
   but it means a new `Pipeline(settings, run_id=...)` must be constructed per
   city. The CLIP singleton lives in the classifier module-level cache, not on
   the Pipeline instance, so constructing N Pipeline objects does not reload
   the model N times.

3. `init_db()` is called in `__init__`. Calling it repeatedly on the same
   `db_path` is safe (SQLAlchemy `create_engine` + `create_all` is idempotent).

**Proposed loop in `__main__.py`:**
```python
for city in city_list:
    city_run_id = f"{city.lower().replace(' ', '-')}_{year}"
    pipeline = Pipeline(settings, run_id=city_run_id)
    stats = asyncio.run(pipeline.run(query=f"{city} {year}", ...))
```

### run_id namespace

Yes — each city should get its own run_id. The `pipeline_run_id` column in
`canonical_cases` and the output JSON filename both use it. Suggested format:
`{city_slug}_{year}` e.g. `arraba_2026`, `sakhnin_2026`. If `--run-id` is
also supplied alongside `--cities`, treat it as a prefix: `arraba_2026_{run_id}`.

## B. CLI shape for stage 5 (query-aware city filter)

### Where in the pipeline does the filter slot?

After merge, before sanity — specifically in `_dedup_and_merge()` after the
`merger.merge_cluster()` call but before the case is appended to `cases`. The
merged `CanonicalCaseSchema` has `case.city` and `case.city_normalized`
populated at that point. The filter should happen here:

```python
# pipeline.py line ~648, inside the for group in all_groups loop:
case = merger.merge_cluster(cluster_input, pipeline_run_id=self.run_id)
if self._city_filter and not self._city_matches(case):
    log.info("city_filtered_out", city=case.city)
    continue
```

The alternative — filtering after merge but before sanity in `_run_cleanup` —
works but breaks the stats: `self.stats["cases_exported"]` is set in
`_dedup_and_merge()` and would be inflated before the filter runs.

An even earlier filter (after extract, before dedup) is tempting but wrong:
the extraction `city` field is free-text from the LLM; the canonical
`city_normalized` dict from the merger is more reliable because `sanity_pass`
already corrected script mixing and ran the gazetteer lookup.

### Failure mode when gazetteer doesn't know the city

**Keep, never drop.** A gazetteer miss means the city is undercatalogued, not
that the case is wrong. The filter should fall back to romanized string
comparison (the same two-level logic already in `is_same_incident()`). Only
drop if the romanized forms are clearly different AND both are non-empty. If
the case has no city field at all, keep it and flag `city_filter_unverified`.

### Opt-out vs opt-in

Opt-in via `--strict-city` is the safer default. During city-mode backfill the
operator typically wants the filter active, but a broad query like
`"Galilee 2026"` that spans multiple cities should not lose valid cases. The
filter being opt-in also means existing scripts that don't use `--cities` are
unaffected. Behavior:

- `--cities X` without `--strict-city`: run N queries but do not post-filter
  results by city (broadens recall).
- `--cities X --strict-city`: run N queries AND drop merged cases whose city
  normalizes to something other than X.

## C. Cross-cutting issues

### Will dedup incorrectly cross-merge cases from different cities?

No, and here is why. The dedup blocking step in `Deduplicator` blocks on
victim name similarity (Jaro-Winkler pre-filter). Two victims in Arraba and
Sakhnin with different names will never form a pair. Two victims with the same
name in different cities will pass the Jaro gate but should fail the cosine
gate (cosine of their article text will be low if the articles discuss
different incidents in different cities). The real risk is if two men named
"Ahmad Jabarin" are killed in two different cities in the same year — but
that is a fundamental limitation of name-based dedup, not a per-city loop
concern. The pipeline does not enforce a city-blocking gate in dedup itself;
the cosine similarity on full article text is the only cross-city guard.

**However**, when running 30 cities against a single `pipeline.db`, the
`raw_articles` table accumulates all fetched articles regardless of run_id.
The `get_all_extractions()` call (used when `extract` stage is skipped) pulls
ALL extractions, not just those from the current run. This means a resume run
(`--stage dedup --stage merge`) on city Arraba will dedup against all
previously fetched articles from Sakhnin, Umm al-Fahm, etc. The fix is to
scope `get_all_extractions()` by `pipeline_run_id` or add a
`--no-accumulate-db` flag that uses a per-city DB path. This is the most
dangerous cross-city bug.

### Output: one envelope per city or one combined?

One envelope per city. The existing `JSONExporter.export_run()` writes
`{output_dir}/{run_id}.json`. With per-city run_ids, you get `arraba_2026.json`,
`sakhnin_2026.json`, etc. — exactly what you want for city-specific analysis.
A combined envelope is a separate consumer concern (a merge script over the per-city
outputs) and should not be baked into the pipeline.

### Gazetteer coverage for Arab-society cities

The gazetteer has **47 cities** as of the current file. It covers the major
Arab-society localities in the Northern (Galilee, Jezreel Valley), Haifa, and
Central districts, and several Negev Bedouin towns. Each entry has aliases in
all three scripts plus romanization variants. The coverage is solid for the ~30
highest-incident cities but will miss smaller villages (e.g. Deir al-Assad,
Fassouta, Aramshe). For those, the romanized fallback in `is_same_incident()`
will carry the filter. Before running a large backfill, it is worth running
`list_all_cities()` and checking which cities in the target list are missing
from the gazetteer — those should be added first.

## D. Risks the architect would miss

### Pipeline.__init__ is heavy and side-effectful

`Pipeline.__init__` does four things that matter for a loop:

1. `init_db(str(settings.db_path))` — idempotent, safe.
2. `MediaPipeline(self._media_settings)` — creates the `MediaClassifier`
   instance which lazy-loads CLIP on first classify call. The CLIP weights
   (~600 MB) are cached in the classifier module's `_model` / `_transform`
   globals, so constructing 30 `Pipeline` objects loads the model exactly once.
   **Safe.**
3. `self.stats` dict initialized to zeros — safe per-instance.
4. `self.run_id` generated — safe, you supply it via `run_id=city_run_id`.

One subtle issue: `asyncio.run()` creates and tears down an event loop per
call in the `__main__.py` loop. Playwright (used by the panet scraper) installs
signal handlers; repeated event-loop creation may produce `RuntimeError:
This event loop is already running` if called from inside a running loop, or
ResourceWarning on Windows if the loop is not cleaned up. Since the loop runs
sequentially (not concurrently), `asyncio.run()` per city is safe on Windows.

### SQLite DB growth over a year-long backfill

With 30 cities x 365 days / 30-day window = 365 runs, each fetching up to
50 articles per source and storing full `raw_html` in `TEXT` columns, the
database grows proportionally. `raw_html` is the dominant cost: a typical
Arabic news article is 30–80 KB of HTML. At 30 cities × 50 articles × 55 KB
average, a single pass accumulates ~83 MB of raw HTML. After a year of weekly
runs, that is ~4.3 GB in a single SQLite file. SQLite handles this fine but
WAL mode (already enabled in `storage/db.py`) is essential for concurrent
readers. Consider adding a `--prune-raw-html` flag that NULLs out
`raw_articles.raw_html` after successful extraction to keep the DB lean.

### The `get_all_extractions()` scoping bug (the real cross-city risk)

As noted in section C, `get_all_extractions()` in `storage/repository.py`
returns ALL extraction rows regardless of `pipeline_run_id`. When the
`extract` stage is skipped (resume mode), the dedup stage picks up every
extraction ever stored. In a 30-city run this means city N's dedup sees N×50
articles instead of 50. This will (a) slow down embedding by N× and (b) risk
false merges on common names. The fix: add a `run_id` filter to
`get_all_extractions()` and pass `self.run_id` from the pipeline. This is a
pre-existing bug that stage 4 will make unavoidable.

### Embedder and name-normalizer module globals

`dedup/embedder.py` lazily loads `paraphrase-multilingual-MiniLM-L12-v2` on
first encode call. Like CLIP, it is a module-level singleton. Running 30
cities sequentially reuses the loaded model — no reload. Running cities in
parallel (e.g. with `asyncio.gather` or `multiprocessing`) would share the
embedder only within the same process; across processes each worker loads its
own copy (870 MB). Sequential is correct here.

## E. Concrete signature proposals

### Stage 4 — `--cities` backfill

```
--cities CITY1,CITY2,...
```

**Behavior contract:** For each city in the list, construct a fresh `Pipeline`
instance with `run_id = {city_slug}_{year}` and call `pipeline.run()` with
`query = "{city} {year}"` and the shared date range / sources / stage flags.
Each city produces its own output JSON. Mutually exclusive with `--query`.

**Example:**
```bash
python -m crime_pipeline \
    --cities arraba,sakhnin,umm-al-fahm \
    --date-from 2026-01-01 --date-to 2026-12-31 \
    --sources ynet,arab48
```

### Stage 5 — `--strict-city`

```
--strict-city
```

**Behavior contract:** Only active when `--cities` is also set. After each
cluster is merged into a `CanonicalCaseSchema`, drop the case if
`case.city` / `case.city_normalized` does not resolve via the gazetteer (or
romanized fallback) to the city currently being queried. On gazetteer miss,
keep the case and add `"city_filter_unverified"` to `case.flags`. A no-op
when `--cities` is absent.

**Example:**
```bash
python -m crime_pipeline \
    --cities arraba \
    --date-from 2026-01-01 --date-to 2026-12-31 \
    --strict-city
```
